## 🧩 3. Стандарты кодирования

### 3.1 PEP8

Следуем стандарту [PEP8](https://peps.python.org/pep-0008/)

**Основные правила:**
- Отступ — **4 пробела**
- Имена переменных и функций — `snake_case`
- Имена классов — `PascalCase`
- Константы — `UPPER_CASE`
- Импорты сгруппированы и находятся в начале файла
- Между функциями — две пустые строки

---

### 3.2 Type Hints

Используются аннотации типов для всех функций:

*def get_student_by_id(student_id: int) -> Student:*

---

### 3.3 Docstrings

Каждый класс и функция должны иметь описания в формате:

*def record_audio(file_path: str, duration: int) -> bool:*
    *"""*
    *Запускает запись аудио и сохраняет файл.*
*
    *Args:*
        *file_path (str): Путь к файлу.*
        *duration (int): Длительность записи (сек).*
*
    *Returns:*
        *bool: True, если запись прошла успешно.*
    *"""*

## 🧩 4. Стандарты кодирования

### 4.1 Основные ветки

| Ветка      | Назначение                                                                                    |
| ---------- | --------------------------------------------------------------------------------------------- |
| **`main`** | Стабильная ветка для демонстрации заказчику и релизов. Вливается только протестированный код. |
| **`dev`**  | Основная ветка разработки. Все задачи выполняются и вливаются сюда после ревью.               |
| ---------- | --------------------------------------------------------------------------------------------- |


### 4.2 Создание новой ветки задачи

dev-<название_задачи_коротко>-<дата>

**Пример**
*git checkout dev*
*git pull origin dev*
*git checkout -b dev-audio-recording-2025-11-12*

### 4.3 Работа с задачей

1. Выполни задачу в своей ветке.

2. Сделай коммит и отправь изменения:
*git add .*
*git commit -m "Added audio recording module"*
*git push origin dev-audio-recording-2025-11-12*

### 4.4 Pull Request

После завершения работы создаётся Pull Request (PR) из твоей ветки в dev.
Оповестить команду о Pull Request.

**Правила:**
- Название PR — краткое и понятное (например, Added protocol generation service)
- В описании указать, что было сделано
- PR должен получить approve от всей команды
- После этого можно делать Merge в dev

### 4.5 Слияние с main

Когда проект готов к демонстрации или релизу:

1. Тестируется вся ветка dev
2. После успешного тестирования делается Pull Request в main
3. После merge в main формируется релиз (версия, changelog)
